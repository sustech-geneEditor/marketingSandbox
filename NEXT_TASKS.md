# 真实多智能体沙盘接线计划

## 目标

把当前可视化网站从“能展示沙盘过程和前端模拟交互”，补到“能从网页真正驱动 Python 沙盘跑真实模型搜索”。

最终要跑通这一条链路：

```text
启动前端和本地后端
-> 网页填写 API Key、Base URL、模型名和沙盘参数
-> 点击真实运行
-> Python MarketingSandbox 调用 Decision / Consumer / Feedback / Critic
-> 网站实时显示发言、策略变化、reward、UCB 和轮次状态
-> 用户可停止、后端可存档、下次可导入并从安全边界续跑
```

## 当前基线

现在已经有的底座：

1. Python 沙盘核心已经存在，包含决策者、消费者、反馈汇总、批评者、动作集和 UCB 搜索控制。
2. `WebRunSession` 已经能描述网页发起的一次真实运行配置。
3. `WebSandboxRunner` 已经能把 Web 会话配置装配成 `MarketingSandbox`，并且已经能默认使用 OpenAI-compatible 真实模型 backend factory。
4. `WebRunEventStream` 已经能把沙盘轮次转换成前端可消费的事件，并支持 JSONL / SSE 风格输出。
5. `WebRunLifecycleManager` 已经有停止信号、安全轮次边界、存档、列档、读档和恢复帮助逻辑。
6. 前端已经有 API 配置入口、真实运行按钮、实时数据展示、前端停止流程、前端本地存档和续跑交互。
7. `WebSandboxApi` 已经把本地 `/api/sandbox/*` 路由接到真实事件流、停止、存档、恢复计划和 provider-check。
8. `web_server` 已经能用标准库启动本地后端，Vite proxy 已经能把前端 `/api` 转到后端。

现在真正缺的部分：

1. 前端 provider-check、真实运行、后端存档和后端恢复计划还需要进一步收紧成完整交互状态。
2. 从真实存档继续搜索还需要把安全边界状态恢复进后续 UCB 搜索，而不是只返回续跑计划。
3. 还没有用真实 API 做完整端到端烟测，确认 provider-check、真实运行、事件流、停止和存档一起跑通。
4. 还没有用真实 API 做端到端网页验收。

## 总体方案

### 架构选择

1. 前端继续保留 Vite + Three.js 可视化站点。
2. 增加一个本地 Python Web 后端，负责运行沙盘、调用模型、推送事件、停止任务、管理真实存档。
3. 前端只通过 `/api/sandbox/*` 调本地后端，不直接把沙盘逻辑搬进浏览器。
4. 第一版模型适配器优先支持 OpenAI-compatible API：`api_key + base_url + model`。这样可以接多个兼容提供商，后续再加专用适配器。
5. 每个角色允许独立模型配置，但先保证一个共享提供商配置能跑通整条链路。
6. 事件流优先用 SSE 或逐行 JSON 流；前端只根据事件更新画面，不直接猜测沙盘内部状态。

### 安全边界

1. API Key 只在当前真实运行请求和后端内存中使用。
2. API Key 不进入事件流、前端存档、后端存档、日志、导出文件和错误提示。
3. reward 和 UCB 是搜索过程指标，不把它包装成真实市场销量预测。
4. Consumer、Feedback、Critic 仍然以文本反馈为主，不为了图表强迫大模型编造概率和评分。
5. 停止只在安全轮次边界生效；没有完成的一轮不能恢复成合法 UCB 历史。
6. 从存档续跑时必须重新提供 API 凭证。
