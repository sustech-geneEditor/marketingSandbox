# AI 营销策略沙盘：满血版设计总结

## 1. 一句话定义

这个沙盘不是让一群大模型简单回答“喜不喜欢这个广告”，而是要搭建一个 **AI-assisted marketing decision sandbox**：

> 在给定目标市场、预算约束和竞争环境下，让不同的合成消费者代理对不同营销策略组合给出结构化反馈，再由策略搜索器比较候选方案，找出在核心人群中同时具备首次转化、复购潜力、竞争韧性和执行可行性的营销策略。

它的价值不在于直接替代真实市场调查，也不在于声称“真实消费者一定会这样买”，而在于：

- 帮助比较多个营销备选方案。
- 暴露不同人群对同一策略的反应差异。
- 找出购买、复购和竞争防御中的关键阻力。
- 把营销决策从“拍脑袋”变成“有备选、有比较、有解释”的过程。
- 为营销计划报告、展示和 Q&A 提供更清楚的策略选择逻辑。

---

## 2. 这个沙盘要解决的核心问题

原始问题可以表述为：

> 什么营销参数组合最能让不同人群购买我的产品？

更完整、更适合沙盘的版本是：

> 在给定目标市场、预算约束与竞争环境下，哪一种营销策略参数组合能在核心细分人群中同时提升首次尝试、复购潜力与竞争韧性，并且能够被清楚解释和执行？

这个表述比“找购买率最高的参数”更好，因为营销策略不能只看短期首购，还要考虑：

- 品牌定位是否一致。
- 折扣和补贴是否可承受。
- 留存机制是否成立。
- 面对竞品时是否容易被抢走。
- 策略是否能在三年预算或 P&L 中落地。

---

## 3. 与 MIS202 营销计划作业的关系

这个沙盘可以成为营销计划中的 **决策支持模块**，但不应该喧宾夺主。

营销计划的主线仍应是：

1. 营销环境分析。
2. 目标客户与消费者洞察。
3. 市场细分、目标市场选择与定位。
4. 营销目标。
5. 多个可行策略备选。
6. 最终营销策略和 marketing mix。
7. 实施计划、预算或三年 P&L。
8. 风险、限制和应对方案。

沙盘最适合支撑下面几件事：

### 3.1 支撑备选方案比较

不要只展示一个最终方案。可以把沙盘用于比较例如：

- `Strategy A`：高折扣快速试用。
- `Strategy B`：品质信任定位 + 精准渠道。
- `Strategy C`：社交传播 + 会员留存。

### 3.2 支撑最终选择理由

最终结论不应只是：

> 我们觉得 B 更适合目标客户。

而应尽量形成这样的表达：

> A 在价格敏感尝鲜人群中首购反应更强，但过度依赖促销，在竞品降价场景下缺乏稳定性；B 在核心目标人群中的信任、复购潜力和竞争韧性更好，且预算压力较低，因此更适合作为主策略。

### 3.3 支撑风险分析和 Q&A

沙盘可以提前回答：

- 为什么不做更大折扣？
- 为什么选这个渠道而不是另一个渠道？
- 哪些人群不适合作为主目标？
- 竞品如果降价或模仿卖点怎么办？
- 这个策略为什么能放进预算和三年经营逻辑？

## 4. 沙盘的总体结构

满血版沙盘可以分成六层：

| 层级 | 核心对象 | 作用 |
|---|---|---|
| 市场层 | 产品、品牌、竞品、预算、外部环境 | 给策略测试提供现实边界 |
| 人群层 | Consumer agents / persona agents | 代表不同细分人群的反应 |
| 策略层 | 营销动作集与参数空间 | 定义可以搜索和比较的策略 |
| 场景层 | 竞争、预算、渠道、口碑等情境卡 | 测试策略在不同压力下的表现 |
| 反馈层 | 消费者反馈归纳、策略感觉、约束和风险 | 判断策略现在强在哪、虚在哪 |
| 搜索层 | `UCBSearchController`、`RewardMapper`、`DecisionAgent` | 先选策略家族，再生成具体策略并保存搜索轨迹 |

完整闭环可以理解为：

```text
市场证据
  -> 人群卡与场景卡
  -> 策略动作集
  -> UCBSearchController 选择本轮 StrategyFamily
  -> DecisionAgent 在被选家族中提出策略候选
  -> Consumer agents 给结构化反应
  -> FeedbackSynthesizer 汇总反馈与策略感觉
  -> CriticAgent 给出边界与自欺风险
  -> RewardMapper 把受控定性标签映射成内部搜索 reward
  -> UCBSearchController 更新 family 搜索状态
  -> 验证集场景复测 top strategies
  -> 输出最终策略、备选方案、风险和解释
```

## 5. 关键角色设计

### 5.1 Consumer agents：合成消费者代理

当前设想是使用不同的 `DeepSeek-V4-Pro` 消费者代理来代表不同人群。

更准确的说法应是：

> 每个消费者代理代表一个经过市场细分定义的 segment archetype，而不是一个真实消费者样本。

这意味着它们适合用于：

- 相对比较策略差异。
- 发现某类人群可能被什么打动。
- 发现某类人群为什么拒绝购买。
- 测试策略在竞品和复购情境下是否稳定。

但不适合直接宣称：

- “真实市场中 73.2% 的消费者会购买。”
- “这个模拟结果等同于一手调研。”

### 5.2 搜索层：策略家族控制器 + 决策模型

当前代码把“搜索”和“出招”拆开了：

- `UCBSearchController` 负责在策略家族之间处理探索与利用。
- `DecisionAgent` 负责在被选中的策略家族里生成具体营销策略。
- `RewardMapper` 负责把总结层和批评层的受控定性标签映射成内部搜索 reward。

这个拆法保留了原来“策略搜索器”的目标，但不让一个大模型同时承担：

- 选择下一轮该探索哪类赢法。
- 在动作空间里写具体营销方案。
- 给自己打 reward。

#### 出招模型可以和消费者模型不同

`DecisionAgent` 的 backend 完全可以和消费者代理使用不同模型，而且这样通常更合理。

- 消费者代理和策略生成器的职责不同。
- 可以减少“同一个模型出题、同一个模型偏爱自己答案”的问题。
- 决策模型可以更偏规划、推理、比较和约束管理。
- 消费者代理可以保持相对统一，方便横向比较。

一个合理的分工是：

| 角色 | 可选实现 | 主要任务 |
|---|---|---|
| Consumer agents | DeepSeek-V4-Pro | 代表人群给出消费反应 |
| DecisionAgent backend | 另一个模型 | 提策略、改参数、解释迭代 |
| UCBSearchController | 确定性搜索控制器 | 选择本轮要探索的策略家族 |
| RewardMapper | 确定性映射器 | 把受控定性标签变成内部搜索信号 |
| FeedbackSynthesizer | 独立总结角色 | 汇总消费者反馈，写出整体策略感觉 |
| Critic agent | 可选独立模型 | 挑漏洞、查不现实假设 |

#### 搜索控制器不一定非得是 LLM

如果营销动作已经被定义成结构化参数，搜索控制也可以部分或全部由传统优化方法承担，例如：

- 网格搜索。
- 随机搜索。
- 贝叶斯优化。
- 遗传算法。
- 多臂老虎机。
- 简单强化学习。

当前实现就是混合式的第一步：

> `DecisionAgent` 负责提出有语义的策略候选和解释；family-level UCB 负责控制探索与利用；消费者反馈与固定约束负责暴露策略差异和风险。

### 5.3 FeedbackSynthesizer：反馈总结层

反馈总结层不负责给策略打一个假精确总分。

更稳妥的原则是：

> 决策者给数值型营销动作；消费者给定性反应；反馈总结者把一批反应写成能推动下一轮决策的策略感觉。

反馈总结层应负责：

- 汇总不同人群反馈。
- 汇总不同场景下的感觉变化。
- 说明哪些人被打动，哪些人没被打动。
- 说明策略最强的地方和最虚的地方。
- 结合预算、执行和品牌边界指出下一轮更值得验证的方向。

### 5.4 Critic agent：反方与质检角色

满血版可以加入一个反方角色，专门质疑策略：

- 这个结果是不是靠大折扣堆出来的？
- 这个策略是否真的符合目标人群？
- 是否存在过度乐观的渠道假设？
- 哪些结论来自市场证据，哪些只是模型推断？
- 哪些重要人群或竞争动作被忽略了？

后续还可扩展：

- `Competitor agent`：模拟竞品如何反击。
- `Finance checker`：检查预算、毛利和 P&L 逻辑。
- `Brand manager critic`：检查定位是否被伤害。
- `Operations checker`：检查渠道、履约、售后是否可执行。

## 6. 消费者人群设计

消费者代理不能只用年龄、收入和身份标签定义。

“大学生”“白领”“年轻女性”这种标签本身不足以指导营销决策。  
更有用的是把人群定义成有动机、有阻力、有替代方案的细分角色。

### 6.1 每个人群卡建议包含的字段

| 字段 | 说明 |
|---|---|
| Segment name | 人群名称 |
| Core need | 他真正要解决的问题 |
| Purchase motivation | 省钱、方便、品质、情绪、社交、身份等 |
| Current alternative | 现在用什么替代方案 |
| Budget level | 预算水平 |
| Price sensitivity | 对价格有多敏感 |
| Trust sensitivity | 对品牌可信度有多敏感 |
| Promotion sensitivity | 是否容易被促销打动 |
| Channel preference | 偏好的购买渠道 |
| Social influence | 是否依赖朋友、评论、KOL、社交证明 |
| Main barrier | 购买最大阻碍 |
| Repeat trigger | 什么会触发复购 |
| Switching threshold | 什么会让他换去竞品 |

### 6.2 更有用的人群切法

同样都是“年轻用户”，也不该只靠年龄标签切。  
当前沙盘默认用 `10` 个可编辑的人群覆盖 archetype 开局：

| 默认 persona | 主要所求利益 / 阻力面 |
|---|---|
| 预算价值务实者 | 这笔支出是否真解决问题 |
| 优惠触发尝鲜者 | 轻承诺入口能否把好奇推成试用 |
| 信任风险谨慎者 | 证据、保障和后悔风险 |
| 便利省时行动者 | 发现、购买、等待和学习摩擦 |
| 习惯锚定复购者 | 旧习惯、重复触发和切换理由 |
| 结果表现优化者 | 具体任务和结果证据 |
| 新鲜体验探索者 | 新体验是否真有记忆点 |
| 口碑比较依赖者 | 相似人群验证和社会证明可信度 |
| 身份意义表达者 | 自我形象、价值和群体归属 |
| 场合关系购买者 | 礼物、共享和社交场合是否合适 |

这 `10` 个不是顶刊给出的固定人格名单。  
它们是沙盘的默认覆盖目录：先按 benefit / job 暴露不同阻力，再用场景卡
施加竞品、信任、预算、渠道和社交压力。

这种切法比单纯的人口统计标签更能指导：

- 定位。
- 定价。
- 渠道。
- 传播卖点。
- 会员和复购机制。

### 6.3 人群卡应该尽量来自市场证据

不要让 persona 完全凭空编造。

可以让人群卡连接到：

- 行业报告。
- 平台评论。
- 竞品评论。
- 公开消费趋势。
- 观察记录。
- 现有二手资料。
- 课堂或项目中已经完成的市场分析。

这样前半部分的环境分析和后半部分的策略沙盘会连起来。

### 6.4 Consumer agents 的行为学底层设定

行为学设定的目标不是把消费者代理变成一个“心理学偏差清单”，而是给它一组更接近真实消费判断的底层先验。

总体原则是：

> 消费者不是完全理性的效用最大化器。它会受注意力限制、当前情境、已有参照点、风险和损失感、即时摩擦、社会影响以及习惯惯性影响。

这些设定应该作为消费判断的背景，而不是强迫每次回答都套上某个理论标签。  
每个消费者代理仍应优先基于：

- 自己的人群画像。
- 当前产品和品牌信息。
- 当前竞争场景。
- 当前营销策略。
- 自己已有的消费习惯和替代方案。

#### 固定注入的行为学先验

| 行为学先验 | 在沙盘中的含义 | 询问时要逼出来的判断 |
|---|---|---|
| 有限注意力 | 消费者不会完整处理所有信息，会先抓最显眼和最相关的线索 | 第一眼注意到什么，忽略了什么，是否看懂 |
| 启发式判断 | 信息不足时会用熟悉度、价格、评论、品牌感、朋友推荐等线索做粗判断 | 哪个线索替代了完整比较 |
| 参照点比较 | 价格和价值会与心理预期、旧习惯、当前替代品和竞品比较 | 他拿什么作为比较对象 |
| 损失与风险敏感 | 买错、踩坑、麻烦、后悔和售后不确定会压低行动意愿 | 最大感知风险和最大后悔点 |
| 即时摩擦与现在偏好 | 现在要花的钱、时间、等待、注册、学习和切换成本会影响转化 | 最大行动摩擦是什么 |
| 社会影响 | 评论、朋友、KOL、群体身份、流行度或专业背书会改变信任 | 哪种社会证明对该人群有效 |
| 习惯与惯性 | 满意不等于复购，旧习惯也会阻碍切换 | 复购触发点和旧习惯阻力 |
| 心理账户 | 涉及价格、套餐、促销和会员时，消费者会判断这笔钱属于哪类预算 | 这笔消费在他心里是必需、奖励、尝鲜还是负担 |

#### 按场景开启的行为学模块

下面这些倾向不建议每次都强塞进消费者询问，而应根据策略和场景开启：

- 稀缺与限时促销。
- 从众热度和爆款信号。
- 身份表达和群体归属。
- 冲动购买和情绪状态。
- 礼物、社交和面子消费。
- 选项过多造成的犹豫。
- 高切换成本和沉没投入。

这样可以避免所有消费者代理都被预设成：

- 看到倒计时就焦虑。
- 看到热门就跟风。
- 看到折扣就冲动。
- 看到会员就愿意长期绑定。

#### 使用边界

行为学先验应帮助沙盘暴露真实阻力，而不是替代市场证据。

- 不能因为有行为学设定，就把合成反馈当作真实调研。
- 不能让搜索层或 `DecisionAgent` 修改这些底层规则来迎合自己的策略。
- 不能用“行为偏差”解释所有不购买行为；有时产品、价格、渠道或定位本身就是不合适。

### 6.5 行为学先验的研究依据

当前 `ConsumerAgent` prompt 注入的不是随手编的“心理学彩蛋”，而是一组克制的行为研究启发。  
这些研究给的是判断先验，不是让合成消费者机械复现论文实验。

详细论文备忘见 `consumer_behavior_literature_notes.md`。

| 研究入口 | 对当前消费者代理的启发 | 在 prompt 里落到哪里 |
|---|---|---|
| Simon 的 bounded rationality | 人在资源和处理能力受限时不会像全知优化器一样决策 | 有限注意力、先看最显眼和最相关线索 |
| Tversky 与 Kahneman 的 heuristics 研究 | 不确定时会借助启发式线索，而不是完整穷举比较 | 熟悉度、价格线索、评论、品牌感与替代方案 |
| Kahneman 与 Tversky 的 prospect theory | 判断会受参照点和损失感影响，潜在损失常比同量收益更刺眼 | 参照点比较、风险、后悔和买错感 |
| Thaler 的 mental accounting | 消费者会给支出做心理分账，不同账户会改变交易感觉 | 价格、套餐、促销、会员与预算感觉 |
| Samuelson 与 Zeckhauser；Wood 与 Neal | 现状偏好与习惯会让切换和复购不只取决于满意度 | 旧习惯、惯性、复购触发与切换阻力 |
| Muchnik、Aral 与 Taylor | 社会反馈本身会影响后续评价和判断 | 评论、朋友、KOL、热度与专业背书只在 persona 合适时生效 |
| Iyengar 与 Lepper | 更多选项不总是更好，过多选择可能让行动变弱 | “选项过多造成犹豫”只作为场景模块开启 |
| Haley 的 benefit segmentation | 人群卡先围绕顾客从产品中寻找什么利益，而不是空人口标签 | 默认十人 persona 的 coverage 轴 |
| Dickson；Belk 的 person-situation / situational 研究 | 同一人群在不同任务、时间、社交和竞争情境下会改变反应 | `Persona` 与 `Scenario` 双层测试 |
| Wind 的 segmentation 研究 | 分群设计要服务决策问题和证据解释 | 默认目录可编辑，不冒充真实市场 census |

当前文档和代码采取的保守做法是：

- 固定注入基础先验。
- 把稀缺、爆款、身份表达、选择过载等保留成场景模块。
- 要求消费者说明这次真正起作用的行为驱动，而不是把所有理论标签都贴一遍。

## 7. 营销策略动作集设计

动作集定义了搜索层和 `DecisionAgent` 能探索什么。

如果动作集一开始太大，搜索器很容易得到一个看起来全面、实际上难解释的混合方案。  
因此第一版要收敛，满血版再扩展。

当前动作集借鉴 marketing mix、品牌意义、顾客旅程和关系营销研究。  
论文采用记录见 `action_space_literature_notes.md`。

### 7.1 文献借鉴后的动作集基线

当前实现继续保留六个顶层动作类别，不把 customer journey、service 或 7Ps 直接硬塞成新 category：

- marketing mix 负责保留 Product、Price、Place/Channel、Promotion 的基本控制面。
- positioning 单列，避免策略只会调战术按钮，却说不清顾客先把它理解成什么。
- retention 单列，避免搜索只追首购，不管购买后关系、复购触发和流失防守。
- 顾客旅程与服务逻辑进入各类子按钮：触点角色落进 `channel`、`promotion`、`retention`，使用支持和保障落进 `product`。

| 动作类别 | 典型参数 |
|---|---|
| 定位参数 | 价值焦点、目标使用场景、比较参照、希望形成的品牌联想、支撑证据角度 |
| Product 产品参数与 offer | 核心价值、功能卖点、品质层级、规格/SKU、包装角色、产品组合、试用、上手支持、服务和保障 |
| 价格参数 | 标准价、入门门槛、折扣率、券、套餐或组合优惠、价格政策、首购/回购激励、价值表达 |
| 渠道参数 | 发现触点、比较或信任触点、转化触点、履约/可得性、售后触点、线上线下或伙伴触点角色 |
| 传播参数 | 主信息、内容形式、传播触点、证据格式、社交证明、试用或激活动作 |
| 留存参数 | onboarding、复购提醒、补货触发、会员/忠诚动作、推荐动作、服务恢复、关系维护 |

这些顶层类别进入当前 `ActionSpace.allowed_categories`。  
更细的按钮应该优先按产品配置到 `parameter_options`，只有价格、折扣、券和预算份额等数值项再配置 `parameter_limits`。

### 7.2 各类动作先问什么

| 类别 | 决策者先回答的问题 |
|---|---|
| 定位 | 顾客应该先把我当成什么价值，与哪个替代方案比较 |
| Product | 我到底交付什么 offer，顾客怎么开始用，风险怎么压住 |
| 价格 | 交易门槛和价值感怎么配，不靠无边界降价糊过去 |
| 渠道 | 顾客在哪里发现、购买、拿到、被支持 |
| 传播 | 用什么信息、证据和触点让定位被看懂 |
| 留存 | 买完之后靠什么关系动作、触发点和服务让他回来或不被带走 |

### 7.3 Product 动作集要展开得更细

在 4Ps 里，`Product` 不应该只写成“卖单个还是卖套装”。  
它要回答的是：

> 我到底给顾客什么东西，它解决什么问题，它用起来是什么体验，它和竞品的差异落在哪里？

Product 动作集可以拆成下面这些层。

#### 7.3.1 核心价值

这是产品最底层的“为什么有人需要它”。

| 可测试方向 | 大白话 |
|---|---|
| 核心问题 | 它帮顾客解决什么麻烦 |
| 使用场景 | 顾客在什么时候、什么地方需要它 |
| 主要收益 | 更省钱、更省时间、更安心、更舒服、更有面子还是更有趣 |
| 目标强度 | 它是刚需、改善型需求，还是尝鲜型需求 |
| 替代对象 | 顾客不用它时会用什么顶上 |

#### 7.3.2 产品本身的功能和卖点

这是产品实际能给出的能力。

| 可测试方向 | 大白话 |
|---|---|
| 关键功能 | 最核心的功能是什么 |
| 卖点优先级 | 先讲哪个好处，后讲哪个好处 |
| 功能丰富度 | 做得简单专注，还是做得更全 |
| 性能表现 | 更快、更稳、更耐用、更舒服、更方便等 |
| 差异功能 | 哪个功能是竞品没有或没讲清的 |
| 使用门槛 | 上手难不难，需不需要学习 |

这里要区分两件事：

- **产品真的具备的功能**，可以拿来测试。
- **产品并不具备的功能**，不能让 `DecisionAgent` 为了反馈好看凭空加上。

#### 7.3.3 品质层级和可信度

很多消费者买的不是“功能列表”，而是对质量和风险的判断。

| 可测试方向 | 大白话 |
|---|---|
| 品质档位 | 入门、主流、中高端还是专业级 |
| 稳定性 | 会不会容易坏、翻车、体验不一致 |
| 安全和合规 | 有没有必须说明的安全、合规或适用边界 |
| 透明信息 | 成分、材质、来源、工艺、参数是否说清 |
| 信任证明 | 检测、认证、质保、真实使用证据等 |
| 风险承诺 | 出问题后怎么退、换、修或补偿 |

#### 7.3.4 规格、版本和 SKU

这是“同一个产品到底怎么切版本”。

| 可测试方向 | 大白话 |
|---|---|
| 容量/尺寸 | 小规格、大规格、家庭装、便携装 |
| 版本层级 | 基础版、升级版、旗舰版 |
| 款式变体 | 颜色、口味、香型、外观、风格等 |
| 人群版本 | 新手版、学生版、专业版、礼赠版等 |
| 使用周期 | 一次性、短周期、长期耐用、可补充 |
| 首次门槛 | 有没有低门槛入门款 |

SKU 不一定越多越好。  
版本太多可能让顾客选择困难，也会增加库存、成本和解释难度。

#### 7.3.5 包装和呈现

包装不是只有“好不好看”，它还会影响理解、信任和使用便利。

| 可测试方向 | 大白话 |
|---|---|
| 包装外观 | 第一眼像便宜、专业、年轻、可靠还是适合送礼 |
| 信息层级 | 顾客能不能快速看懂是什么、有什么好处 |
| 便携性 | 好不好带、好不好放、好不好打开 |
| 使用便利 | 是否容易取用、保存、补充或清洁 |
| 礼赠属性 | 能不能拿来送人，送出去是否体面 |
| 环保和材料 | 是否需要强调少浪费、可回收或耐用 |

#### 7.3.6 产品组合和购买形态

这是 Product 和 offer 最容易交叉的地方。

| 可测试方向 | 大白话 |
|---|---|
| 单品 | 只买一个核心产品 |
| 套装 | 多个产品搭着卖 |
| 新手包 | 第一次买就给一套最容易上手的组合 |
| 试用装 | 先降低第一次尝试成本 |
| 补充装 | 买过后更便宜、更方便地补货 |
| 订阅/周期包 | 适合有规律复购的产品 |
| 礼盒 | 适合节日、送礼或高客单场景 |

#### 7.3.7 产品体验

顾客买到手之后的体验也属于 Product。

| 可测试方向 | 大白话 |
|---|---|
| 开箱体验 | 到手第一感觉好不好 |
| 上手引导 | 有没有说明、教程、示范或 onboarding |
| 使用过程 | 用的时候顺不顺 |
| 使用结果 | 结果是否稳定、明显、符合预期 |
| 维护成本 | 用完后要不要清理、保存、维修或补充 |
| 失败容错 | 第一次没用好时会不会立刻放弃 |

#### 7.3.8 附加服务和保障

很多时候顾客买的是“产品 + 服务承诺”。

| 可测试方向 | 大白话 |
|---|---|
| 售前支持 | 买前能不能问清楚 |
| 配送/交付 | 拿到手快不快、稳不稳 |
| 安装/设置 | 是否需要帮他完成第一步 |
| 售后服务 | 出问题能不能找到人 |
| 退换保障 | 不合适时能不能降低后悔成本 |
| 质保承诺 | 能保多久，保什么 |

#### 7.3.9 Product 里哪些要固定，哪些可以让沙盘搜

Product 最容易被策略生成过程搜歪，所以要先分边界。

**必须先固定的东西：**

- 产品真实能做到什么，不能做到什么。
- 成本、毛利和供应能力的大边界。
- 法规、安全、健康、质量等不能乱承诺的内容。
- 品牌不愿意牺牲的底线。

**适合放进动作集搜索的东西：**

- 哪个核心卖点优先。
- 主推哪个使用场景。
- 做入门款、主力款还是升级款。
- 规格怎么切。
- 要不要加试用装、新手包、补充装或礼盒。
- 包装强调便携、专业、礼赠、环保还是信息清晰。
- 要不要加上手引导、售后保障或退换承诺。

这样 Product 才既足够细，又不会变成“模型为了讨好顾客随便发明新品”。

### 7.4 第一版不要开放太多参数

第一版更建议只开放 `5` 到 `8` 个关键参数。

这样可以回答：

- 到底是价格带来了变化？
- 到底是定位话术带来了变化？
- 到底是渠道便利带来了变化？
- 到底是会员留存机制带来了变化？

如果一轮里所有东西都改了，最后很难解释策略为什么赢。

### 7.5 动作参数分两种

#### 结构化参数

适合程序或传统搜索：

- 折扣率。
- 渠道预算比例。
- 是否加试用装。
- 是否加会员返券。
- KOL 投放强度等级。

#### 语义型参数

适合 LLM 提案：

- 品牌定位表述。
- 广告信息重点。
- 内容主题。
- 竞品应对叙事。
- 体验承诺的表达方式。

## 8. 场景卡设计

消费者不是在真空中决策。  
同一个人群在不同市场场景下会给出不同反应。

### 8.1 建议加入的基础场景

- 正常首发场景。
- 用户预算变紧场景。
- 竞品降价场景。
- 竞品品牌更有信任优势场景。
- 竞品渠道更方便场景。
- 竞品通过朋友推荐或社交传播进入用户视野场景。
- 品牌出现负面评价或信任疑虑场景。
- 促销季信息噪音很大场景。

### 8.2 场景卡的价值

场景测试能把结论从：

> Strategy A 看起来整体最好。

升级成：

> Strategy A 在价格敏感人群中首购强，但在竞品降价时很脆弱；Strategy B 平均首购略低，却在核心人群的信任、复购和竞争场景中更稳。

这种结论更适合真实营销决策，也更容易写进备选方案比较。

## 9. Consumer agents 应该输出什么

最初想到的输出已经包括：

- 第一印象怎么样。
- 会不会买。
- 会不会回头买。
- 遇到竞争品后还会不会买我的。

这些是核心，但还不够。  
如果只看“买不买”，你无法知道问题到底出在：

- 定位不清。
- 不信任。
- 不觉得值。
- 渠道不方便。
- 首购可以但没有复购理由。
- 竞品替代太容易。

### 9.1 按消费者旅程输出

| 阶段 | 建议输出 |
|---|---|
| 看到策略 | 第一眼注意到什么、忽略什么 |
| 理解阶段 | 他认为你卖的是什么、核心卖点是什么 |
| 价值判断 | 是否觉得值、价格是否公平 |
| 首购决策 | 立即买、考虑、不买及原因 |
| 条件变化 | 改什么会让他改变决定 |
| 渠道行为 | 想在哪里买、何时买、是否要试用 |
| 使用后预期 | 期待什么体验、哪里可能失望 |
| 复购判断 | 是否复购、多久复购、靠什么触发 |
| 竞争冲击 | 竞品做什么最容易把他抢走 |
| 传播行为 | 是否推荐、分享、评价、带朋友购买 |

### 9.2 更完整的输出字段

#### 认知和感受

- 第一印象关键词。
- 感知到的品牌定位。
- 最吸引人的卖点。
- 最困惑的一点。
- 最不信任的一点。
- 情绪反应：兴奋、安心、无感、怀疑、压力感等。

#### 购买判断

- 当前态度：愿意尝试 / 继续考虑 / 明显无感。
- 对当前价格和价值感的定性评价。
- 是否像是依赖折扣才会行动。
- 最关键购买触发点。
- 最关键拒绝原因。

#### 行为细节

- 偏好购买渠道。
- 是否愿意试用、小规格购买、订阅或加入会员。
- 是否愿意关注品牌或留下联系方式。

#### 留存与流失

- 复购前提。
- 复购周期。
- 哪些体验会导致不再购买。
- 哪些服务会提升复购。
- 对积分、返券、组合包、提醒机制的反应。

#### 竞争反应

- 竞品更便宜时是否容易切换。
- 竞品更有品牌信任时是否切换。
- 竞品渠道更方便时是否切换。
- 竞品社交传播更强时是否切换。
- 品牌需要提供什么，才不容易被切走。

#### 品牌传播与资产

- 是否会推荐。
- 是否会发社交平台。
- 是否会留下评价。
- 会如何向朋友描述这个品牌。
- 这个策略会让品牌显得廉价、可信、专业、年轻、有趣还是浮夸。

### 9.3 必须强制问失败条件

每个消费者代理最好都回答：

- 哪一点让我不信？
- 哪一点让我觉得不值？
- 哪一点让我懒得行动？
- 哪一个竞品动作最容易把我抢走？
- 改哪一个地方，我会改变决定？

这类负反馈比单纯“喜欢”更能帮助优化策略。

### 9.4 每次消费者询问可复用的行为学设定块

下面这段可以作为每次询问 Consumer agent 时的固定前置设定。  
它定义的是判断方式，不应覆盖 persona、产品信息、营销策略和场景卡。

```md
## 行为学底层设定

你不是一个完全理性的效用最大化器，而是一个真实情境中的消费者代理。
在判断营销策略时，请遵循以下行为倾向，但不要机械套用，也不要为了符合理论而强行制造反应：

1. 你的注意力有限。你会先注意最显眼、最相关、最容易理解的信息，可能忽略复杂细节。
2. 在信息不足或不确定时，你会依赖启发式线索做粗判断，例如熟悉度、价格锚点、评论、朋友推荐、品牌可信度、视觉专业度和当前替代方案。
3. 你会基于参照点比较，而不是只看绝对值。请把当前方案与自己的预期价格、已有购买习惯、竞品和当前替代方案比较。
4. 你会在意损失、风险和后悔。潜在踩坑、麻烦、质量不确定、售后不确定和切换失败可能比同等收益更影响你的决策。
5. 你会受到即时摩擦影响。现在要花的钱、时间、精力、等待、注册步骤、学习成本和切换成本会降低行动意愿；未来收益不一定能完全抵消这些摩擦。
6. 你会受到社会影响，但敏感度取决于你的人设。朋友、评论、KOL、群体身份、流行程度或专业背书可能影响你的信任和选择。
7. 你的复购和竞品切换会受到习惯与惯性影响。满意不等于一定复购；稳定场景、便利性、提醒、触发点和旧习惯都会影响后续行为。
8. 涉及价格、套餐、会员、促销时，请考虑心理账户：你会判断这笔钱属于哪类消费预算，并判断这笔交易是否划算、值得、有负担或像被坑。

回答时请优先基于：
- 你的人群画像
- 当前产品和品牌信息
- 当前营销策略
- 当前场景与竞品环境

不要把上面的行为学倾向当成固定答案。
请明确指出哪些倾向在这次判断中真的起作用，哪些没有明显起作用。
```

### 9.5 每次消费者询问建议增加的行为学诊断输出

除了首购、复购、竞品冲击和推荐意愿，建议固定要求消费者代理给出下面五项诊断：

```md
## 行为学诊断输出

1. 最先注意到的线索：
2. 主要参照点：
3. 最大感知损失或风险：
4. 最大行动摩擦：
5. 本次判断中最重要的行为驱动：
```

这五项会让搜索层和 `DecisionAgent` 看到“不买背后的机制”，而不只是看到一句无感。

## 10. 反馈总结和约束判断

### 10.1 不要把合成反馈压成假精确分

消费者代理不是真实统计样本，反馈总结者也不应凭感觉给出总分、维度分或购买概率。

第一版更适合保留一套固定的反馈观察维度：

| 观察维度 | 要回答的问题 |
|---|---|
| Attention | 顾客第一眼看到了什么 |
| Clarity | 卖点和定位是否被看懂 |
| Relevance | 核心人群是否觉得和自己有关 |
| Trust | 信任是被建立了，还是仍然犹豫 |
| Trial Intent | 首次尝试的感觉是被推近还是被推远 |
| Value Perception | 价格和价值感是否匹配 |
| Retention Potential | 复购理由是否自然 |
| Competitive Resilience | 竞品一来时策略显得稳还是脆 |
| Advocacy Potential | 是否有推荐和传播的感觉 |
| Feasibility | 动作是否像能落地 |
| Brand Risk | 是否伤害品牌定位 |

### 10.2 总结者要写出策略感觉和边界

`FeedbackSynthesizer` 应把消费者反馈总结成文字，例如：

- 这套策略整体像什么打法。
- 哪些人群被打动。
- 哪些人群仍然不信。
- 首购、复购和竞品冲击下各自暴露了什么问题。
- 当前策略是不是太靠折扣、赠品和资源堆砌。
- 下一轮最值得改哪几个动作。

同时要保留固定约束：

- 预算压力。
- 执行复杂度。
- 品牌风险。
- 产品真实边界。
- 法规、安全和供应限制。

这样决策者仍然可以给出数值型价格、折扣和预算动作，但消费者反馈与整体评议不会被伪装成精确市场预测。

---

## 11. 搜索流程建议

### 11.1 训练与验证要分开

为了避免搜索层过拟合少数人群和少数场景，建议把测试拆成两部分。

#### 搜索集

搜索层可以在这部分上反复迭代：

- 主目标人群。
- 常规场景。
- 主要竞品压力场景。

#### 验证集

top strategies 再放到搜索层未直接优化过的条件中复测：

- 新的次级人群。
- 更苛刻的竞品降价。
- 信任危机。
- 渠道触达不足。
- 用户预算下降。
- 卖点被竞品模仿。

如果一个方案只在搜索集好看、验证集崩掉，就说明它不够稳健。

### 11.2 建议的迭代闭环

每轮可以走：

1. 搜索层选择要探索的策略方向，`DecisionAgent` 提出一批策略候选。
2. 策略候选被转成结构化参数和明确的营销表述。
3. Consumer agents 在指定场景卡下给反馈。
4. FeedbackSynthesizer 汇总整体反馈感觉和关键问题。
5. Critic agent 挑出策略漏洞。
6. 搜索层根据聚合反馈控制下一轮探索，`DecisionAgent` 调整候选策略。
7. 当表现趋稳后，保留 top strategies 进入验证集。
8. 最后输出主策略、备选策略、淘汰理由和风险应对。

### 11.3 搜索层看什么反馈更合适

搜索器不一定要看每个消费者的全部长对话。  
更稳的方式是给它看：

- 聚合后的定性搜索信号。
- 各人群的关键拒绝原因。
- 关键场景下的失稳原因。
- 预算和品牌风险提示。
- 代表性原话摘要。

这样它更像在做策略迭代，而不是迎合某几个 persona 的语言偏好。

### 11.4 当前已落地的 family-level UCB 路径

当前代码已经把探索与利用落在策略家族层，而不是让 `DecisionAgent` 自己暗自决定下一轮只修旧方案。

开启 UCB 时，一轮顺序是：

1. `UCBSearchController` 选择本轮 family slots。
2. `SearchBrief` 只给 `DecisionAgent` 家族说明、生成意图和受控文字记忆。
3. `DecisionAgent` 对每个选中 family 生成一个 concrete `Strategy`，并写出 `family_id` 与 `family_fit_note`。
4. `MarketingSandbox` 仍按 persona 与场景测试策略。
5. `FeedbackSynthesizer` 输出总结文字和 `FeedbackSearchSignals`。
6. `CriticAgent` 输出批评文字和 `CriticSearchSignals`。
7. `RewardMapper` 把这些定性标签映射成内部 reward。
8. `UCBSearchController` 更新 family state，并把 trace 存进 `RoundResult` 和 `SandboxResult`。

不开启 UCB 时，原有定性回合流程仍然保留。  
无论哪条路径，消费者代理都看不到 family selection、UCB score 和内部 reward。

## 12. 需要重点防的坑

### 12.1 同模型偏好

如果 `DecisionAgent` 和 consumer agents 使用同一类模型，可能出现：

- 搜索器偏爱某种表述。
- 消费者代理也偏爱同一种表述。
- 最终优化的是模型风格偏好，而不是市场判断。

把 `DecisionAgent` backend 换成另一模型有帮助，但不能完全消除偏差。  
还需要：

- 固定人群卡。
- 固定场景卡。
- 固定反馈观察维度。
- 固定预算约束。
- 做验证集复测。

### 12.2 把合成反馈当真实预测

沙盘应定位为：

- 策略比较工具。
- 假设暴露工具。
- 风险发现工具。
- 决策解释工具。

不要把它包装成：

- 真实调研替代品。
- 精确销量预测器。
- 真实购买率统计来源。

### 12.3 人群定义漂移

不要让搜索层或 `DecisionAgent` 因为结果不好就偷偷改：

- 目标人群定义。
- 购买偏好。
- 场景压力。
- 竞品规则。

否则测试环境会被一起优化掉，结果没有意义。

### 12.4 动作集过大

动作集太大时会导致：

- 很难解释什么因素带来了提升。
- 很难区分营销策略和资源堆砌。
- 很难放进预算与执行计划。

第一版要克制，满血版再扩展。

## 13. 推荐的版本路线

### 13.1 第一版 MVP

第一版目标不是做出最复杂的多智能体社会，而是先跑通营销决策闭环。

建议范围：

- 选定 `1` 个具体产品或服务。
- 准备默认 `10` 个消费者覆盖 persona；具体项目可基于市场证据合并、删改或重写。
- 定义 `3` 个策略家族。
- 每个策略开放 `5` 到 `8` 个关键参数。
- 准备 `4` 到 `6` 个核心场景卡。
- 输出完整消费者旅程反馈。
- 使用固定反馈结构比较策略。
- 保留主策略、次优策略和失败策略。

第一版应该能回答：

- 谁是核心目标人群？
- 什么卖点最能打动他？
- 什么让他不买？
- 首购、复购和竞品冲击下策略表现是否一致？
- 为什么最终选这个方案而不是别的方案？

### 13.2 满血版

在第一版跑通后，可以扩展：

- 更多细分人群。
- 多阶段消费者旅程。
- 更丰富的竞品反击逻辑。
- 多模型交叉验证。
- `DecisionAgent` 与数值优化器或搜索控制器混合。
- 财务检查器和执行检查器。
- 品牌风险检查。
- 多轮营销节奏：launch、growth、retention、defense。
- 多年预算与阶段性 KPI 对齐。

## 14. 满血版输出物

沙盘最终不应只输出一个“最佳参数”。

更好的输出物包括：

### 14.1 策略结论

- 推荐主策略。
- 两到三个重要备选策略。
- 被淘汰策略及淘汰原因。
- 关键参数组合。
- 每个参数改变了什么。

### 14.2 人群结论

- 核心目标人群。
- 次级目标人群。
- 不值得优先投入的人群。
- 各人群最主要购买动机。
- 各人群最主要购买阻力。

### 14.3 场景结论

- 正常场景下的表现。
- 价格战下的表现。
- 信任竞争下的表现。
- 渠道便利竞争下的表现。
- 负面口碑或不确定性下的表现。

### 14.4 商业结论

- 预算压力。
- 执行复杂度。
- 品牌风险。
- 复购逻辑。
- 对三年营销计划的含义。

### 14.5 报告和展示素材

- 备选方案比较表。
- 策略选择理由。
- 风险与缓释动作。
- 关键假设清单。
- Q&A 防守点。

## 16. 当前最值得先定下来的问题

在继续做沙盘前，最重要的不是先写代码，而是先定四件事：

1. 具体营销对象是什么产品、服务或品牌？
2. 本次营销目标优先级是什么？
3. 准备定义哪几类消费者人群？
4. 允许搜索层和 `DecisionAgent` 探索哪些营销动作参数？

再往前一步，还应补定：

5. 主要竞品是谁？
6. 预算和品牌边界是什么？
7. 首版要优化首购、留存还是竞争防御，三者权重如何安排？
8. 哪些市场证据可以支撑 persona 和场景设计？

## 17. 最终建议

这个项目值得做，但它最有价值的形态不是“AI 替我预测市场”，而是：

> 用一套可控的合成消费者、竞争场景、动作空间和反馈归纳逻辑，系统比较营销策略，并把最终营销计划的选择理由讲清楚。

第一版建议采取以下原则：

- 人群少而清楚。
- 参数少而可解释。
- 场景要包含压力测试。
- 搜索层、`DecisionAgent` 与 Consumer agents 分工。
- 策略感觉之外保留失败原因。
- 结论必须能回到预算、定位和执行。

如果做到这些，这个沙盘既可以服务后续实现，也可以服务营销计划报告与展示中的策略论证。
