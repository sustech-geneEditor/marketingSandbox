# 营销多智能体沙盘可视化网站设计

## 1. 一句话定义

这个网站不是把沙盘结果做成一张静态报表，而是把营销策略搜索过程变成一个可以实时观看、暂停、回看和解释的三维沙盘：

> `DecisionAgent` 站在中央不断出招，周围的消费者代理、反馈总结者和批评者轮流回应；网站一边展示他们在这一轮说了什么，一边展示策略家族搜索、动作集变化、风险反馈和当前积累到的证据。

---

## 2. 设计目标

### 2.1 用户真正想看到什么

这个网站首先要回答四个问题：

1. 现在是谁在说话、在反应什么？
2. 当前搜索到了哪一种营销打法，为什么会轮到它？
3. 这套策略在消费者、总结者和批评者那里暴露出了什么信号？
4. 多轮搜索之后，哪些策略方向更值得继续验证，哪些方向看起来不稳？

### 2.2 演示目标

如果拿给老师和同学演示，网站应该做到：

- 不需要先看终端输出就能理解沙盘在运行。
- 一眼能看出这是多智能体互动，不是一个模型单独聊天。
- 一眼能看出这是营销策略搜索，不只是角色扮演。
- 能解释 UCB 在平衡探索和利用，而不是让决策模型一直局部微调。
- 能把“搜索信号”和“真实市场预测”明确区分开。

### 2.3 不做什么

第一版不要把它做成：

- 纯仪表盘。
- 纯聊天记录页面。
- 只显示最终最优方案的结果页。
- 把 UCB reward 包装成购买率、复购率或市场份额预测的页面。
- 视觉特别热闹，但看不出策略是怎么一步步被试出来的页面。

---

## 3. 网站的总体气质

## 3.1 视觉方向

网站整体采用：

> 云端、空灵、发光、温柔、有陪伴感的三维沙盘氛围。

氛围参考《光遇》式的世界感：

- 角色像在一个温和的云上议事场里。
- 光是信息流的一部分，不只是装饰。
- 场景有缓慢呼吸感，不要像压迫性的控制室。
- 观众能感觉一群 agent 正围着一个策略问题讨论。
- 后续主场景背景可以从浅云雾升级成深邃星空，让策略台像漂在可探索的宇宙沙盘里。

但网站不能只像游戏场景。它仍然要保留营销研究和搜索过程的可读性：

- 数据面板要清楚。
- 当前轮次要清楚。
- 当前策略动作要清楚。
- 每个角色的反馈角色边界要清楚。

## 3.2 角色风格

角色不做写实人脸，统一做低细节、抽象、亲和的小人。

建议方向：

- 身体：简化披风、袍形、发光边缘，角色靠轮廓和颜色区分。
- 脸：圆润、低细节、呆萌、好接近。
- 表情：只保留少量抽象表情状态，例如专注、犹豫、赞同、警惕。
- 奶龙感：借鉴圆润、憨萌、友好的感觉，不直接复制现成角色脸。

这样做有两个好处：

- 角色看起来像一群可识别的数字生命，而不是冷冰冰的节点。
- 不同 persona 的差异可以更多放在光色、姿态、标签和说话内容上，而不是硬做复杂脸模。

## 3.3 建议色彩

第一版不要全屏只有同一种蓝紫色。

建议主色分工：

| 用途 | 方向 |
|---|---|
| 场景底色 | 云白、雾青、浅灰青 |
| 中央策略光源 | 暖金、柔白 |
| 消费者代理 | 多组低饱和角色色 |
| 批评和风险 | 珊瑚橙、琥珀红 |
| 搜索轨迹 | 青绿、金色、少量冷紫点缀 |
| 数据文字 | 深石墨、冷灰 |

### 星空背景升级

深邃星空适合放在三维主场景里，不建议直接把左侧配置区、右侧解释区
和正文文字都改成低对比黑底。  
实现时要守住：

- 策略台、十人外圈和中央三角色仍是第一视觉焦点。
- 星星像萤火虫一样缓慢漂移：有轻微游动、远近层次和呼吸感，不是静止噪点贴图。
- 漂移动画要像苹果系界面一样丝滑克制：连续、稳定、低干扰，不闪烁、不顿挫、不突然乱跳。
- 星空有远近层次和缓慢流动，但不能靠大面积炫光抢走泡泡与标签。
- 数据面板保持高可读性；星空只负责沉浸，不负责藏信息。
- 桌面和移动端都要检查角色轮廓、动作标签、对话泡泡是否仍清楚。

## 4. 首屏形态

## 4.0 本地打开指引

当前可视化网站是 Vite + Three.js 前端，不是直接双击 `visualization_site/index.html`
就能完整运行的单文件页面。

- 先在 `visualization_site` 下运行 `npm install` 和 `npm run dev`。
- 再打开 Vite 打印出的本地地址，默认通常是 `http://127.0.0.1:5173/`。
- 只演示示例回放和课堂数据包时，前端服务就够了。
- 点真实运行时，还需要本地后端把 live event、stop 和 archive 接口挂出来。

`index.html` 自身保留一段直开 fallback 指引。  
如果用户直接双击页面导致模块没加载，页面应解释要启动本地网页服务，而不是留一个空白屏。

## 4.1 不是落地页，而是工作台

网站打开后的第一屏就应该是可操作的沙盘工作台。

第一屏不需要先放一个巨大营销式 hero 介绍项目。用户进入后应该直接看到：

- 中央三维沙盘。
- 当前是否已经配置 API。
- 当前是否已经载入产品、人群、场景和动作集。
- 一个明显的“开始搜索”入口。
- 一个能打开设置的入口。

## 4.2 首屏布局

```text
+--------------------------------------------------------------------------------+
| 顶部状态栏: 项目名 | API 状态 | 当前轮次 | 搜索模式 | 开始/暂停/单步/停止/导入   |
+----------------------+--------------------------------------+------------------+
| 左侧配置抽屉         |                                      | 右侧实时解释面板 |
| - API 与模型         |          三维沙盘主场景              | - 当前策略       |
| - 产品与品牌事实     |                                      | - 当前发言       |
| - 人群与场景         |     DecisionAgent 在中央             | - 总结与批评     |
| - 动作集与搜索设置   |     其他 agent 环绕                  | - 搜索轨迹       |
|                      |                                      |                  |
+----------------------+--------------------------------------+------------------+
| 底部时间线: 轮次 | 当前事件 | 已试策略 | family 搜索轨迹 | 回看锚点              |
+--------------------------------------------------------------------------------+
```

## 4.3 布局原则

- 三维场景是主角，占据最大面积。
- 数据面板必须读得清，不把所有内容塞进 3D 世界里。
- 说话泡泡只展示短反馈，详细文本落在右侧面板。
- 左侧配置可以折叠，演示时让场景更开阔。
- 底部时间线让用户知道这不是一瞬间的结果，而是多轮搜索。

## 5. 三维沙盘场景设计

## 5.1 场景构图

场景中心是一张发光的策略沙盘台，或者一块悬浮的圆形策略地台。

围绕关系：

| 对象 | 场景位置 | 视觉含义 |
|---|---|---|
| `DecisionAgent` | 中央 | 负责出营销策略 |
| `ConsumerAgent[]` | 中央外圈环绕，默认先摆十人覆盖环 | 代表不同人群接招 |
| `FeedbackSynthesizer` | `DecisionAgent` 右侧 | 收拢消费者反馈 |
| `CriticAgent` | `DecisionAgent` 左侧 | 专门指出漏洞 |
| `UCBSearchController` | 不一定做成人 | 用策略轨迹、星图或 family 光路表示 |
| `ActionSpace` | 中央沙盘上的六类动作轨道 | 说明决策者能动哪些营销按钮 |

默认十人外圈是 benefit-led 覆盖目录，不是十个固定人口标签。  
网站准备区要允许后续按项目证据改人群卡，场景卡再给这些人群施加不同
购买任务、竞品、信任和摩擦压力。

`FeedbackSynthesizer` 和 `CriticAgent` 不放进消费者外圈。  
它们靠近中央决策者两侧：一个把外圈反馈收拢给决策者，一个从另一侧
拆策略漏洞，这样课堂演示时角色层次更直观。

## 5.2 中央策略沙盘

中央策略沙盘展示当前策略最重要的东西：

- 当前策略名。
- 当前策略所属 `StrategyFamily`。
- 当前策略正在动的动作类别。
- Product、Price、Place、Promotion 等关键动作变化。
- 如果开启 retention 或 positioning，也要一起显示。

建议把动作类做成中央桌面上的发光槽位：

| 槽位 | 展示内容 |
|---|---|
| Positioning | 主打定位、价值焦点 |
| Product | 产品形态、包装、保障、服务支持 |
| Price | 标价、折扣、券、价值解释 |
| Channel | 发现、转化、履约、售后触点 |
| Promotion | 内容、证据、传播方式 |
| Retention | 复购触发、会员、服务恢复、关系维护 |

## 5.3 角色说话机制

当某个 agent 发言时：

1. 角色轻微亮起。
2. 角色面向中央或面向当前策略。
3. 角色右上角出现柔光对话泡泡。
4. 右侧实时面板同步展开该发言的详细版本。

对话泡泡不应该太长。

建议泡泡只放：

- 第一印象一句话。
- 当前态度一句话。
- 风险提醒一句话。
- 复购或竞品冲击的一句短判断。

长反馈放进右侧面板，否则场景会被文本遮住。

## 5.4 不同角色的视觉语言

| 角色 | 视觉状态建议 |
|---|---|
| `DecisionAgent` | 中央暖金光，出策略时桌面符号向外扩散 |
| 消费者代理 | 不同轮廓光和 persona 标签，说话时出现柔光泡泡 |
| `FeedbackSynthesizer` | 多束消费者反馈光线向它汇聚，再形成一段总结光片 |
| `CriticAgent` | 风险出现时有琥珀色警示环，不做攻击性红屏 |
| 输出守门失败 | 页面出现契约错误提示，场景动作暂停，不把错误当正常事件 |

## 6. 网站信息架构

建议分为三个主要工作状态，而不是三个完全割裂的网站页面。

| 状态 | 用户在做什么 | 核心界面 |
|---|---|---|
| 准备 | 配 API、产品、人群、场景、动作集和搜索设置 | 左侧配置抽屉 + 中央待机沙盘 |
| 实时搜索 | 看策略一轮轮生成、被反馈、被批评、被更新 | 三维场景 + 实时解释面板 + 时间线 |
| 回看与比较 | 回放关键轮次，比较策略和 family 轨迹 | 时间线回放 + 策略对比面板 |

## 7. API 与模型配置入口

## 7.1 为什么它必须是网站的一等入口

这个项目有多个模型角色，而且它们不一定要用同一个模型。

所以网站必须让用户看得见：

- 谁在用哪个模型。
- 哪个 API 已经连上。
- 哪个角色还没有配置。
- 本次运行到底是实时调用模型，还是回放示例数据。

## 7.2 放在哪里

建议有两个入口：

1. 首次进入准备状态时，左侧第一个配置卡就是“API 与模型”。
2. 运行页顶部状态栏保留一个 API 状态按钮，点击后重新打开配置抽屉。

这样做的好处：

- 开始前不会忘记配置。
- 演示时出现连接问题也能很快定位。
- 不会把 API 设置埋进开发说明里。

## 7.3 配置结构

建议先分两层：

### 全局提供方配置

| 字段 | 说明 |
|---|---|
| Provider | 选择 API 提供方 |
| Base URL | 兼容不同服务端地址 |
| API Key | 掩码输入 |
| Connection Test | 测试是否能连接 |
| Default Model | 默认模型 |
| Timeout / Retry | 高级运行设置 |

### 角色级模型映射

| 角色 | 建议配置 |
|---|---|
| `DecisionAgent` | 可选强推理模型 |
| `ConsumerAgent[]` | 可统一使用一类消费者模型，也可按 persona override |
| `FeedbackSynthesizer` | 可与消费者模型分开 |
| `CriticAgent` | 可单独用更严格的模型 |

角色级映射界面可以长这样：

| 角色 | 使用模型 | 继承默认配置 | 单独覆盖 |
|---|---|---|---|
| DecisionAgent | `model-a` | 否 | 是 |
| ConsumerAgent group | `model-b` | 否 | 是 |
| FeedbackSynthesizer | 默认 | 是 | 否 |
| CriticAgent | `model-c` | 否 | 是 |

## 7.4 配置体验

API 设置区建议具备：

- 密钥掩码显示。
- 显示“未配置”“已连接”“连接失败”的状态。
- “测试连接”按钮。
- “使用后端环境默认配置”选项。
- “保存模型路由配置但不保存明文密钥”的说明。
- “示例回放模式”入口，方便课堂演示时不用现场等待所有模型请求。

## 7.5 安全边界

如果网站后面有后端，优先遵守：

- 浏览器不直接把密钥暴露给第三方模型提供方调用链。
- API Key 不写进前端代码。
- API Key 不放进公开日志、截图和导出结果。
- 如果保存配置，默认只保存模型路由和非敏感参数。
- 演示环境优先支持后端环境变量或临时 session 密钥。

## 8. 准备区要输入什么

除了 API，网站还要让用户配置沙盘事实。

## 8.1 输入模块

| 模块 | 主要内容 |
|---|---|
| 产品与品牌事实 | 产品能力、边界、品牌背景、预算和目标 |
| 人群卡 | persona 列表、购买动机、阻力、替代方案 |
| 场景卡 | 正常场景、竞品压力、预算压力、信任压力等 |
| 动作集 | 允许改哪些营销按钮 |
| 搜索设置 | 是否开启 UCB、family 列表、轮次、每轮候选数 |
| API 与模型 | 各角色模型调用配置 |

## 8.2 动作集配置要可见

动作集不能只在后台存在。

网站应让用户看到当前允许搜索的动作大类：

- Positioning
- Product
- Price
- Channel
- Promotion
- Retention

每类可以折叠查看详细按钮，但第一屏不必一次展开所有参数。

## 9. 实时搜索页要展示的数据

## 9.1 当前轮次数据

每一轮至少要能看到：

- 当前 round。
- 当前被选择的 `StrategyFamily`。
- 当前生成的具体策略。
- 当前策略改了哪些动作。
- 当前测试了哪些 persona 和 scenario。
- 当前谁正在发言。

## 9.2 消费者反馈

消费者代理不能只显示“买”或“不买”。

界面需要显示他们的定性反馈：

- 第一印象。
- 主要吸引点。
- 主要拒绝点。
- 当前消费态度。
- 复购感觉。
- 竞品压力下的变化。
- 关键行为学诊断，例如先注意到什么、主要参照点是什么、最大风险和最大行动摩擦是什么。

## 9.3 总结与批评

`FeedbackSynthesizer` 区域展示：

- 整体策略感觉。
- 哪些人群被打动。
- 哪些人群没有被打动。
- 当前强点。
- 当前短板。
- 下一轮值得继续验证什么。

`CriticAgent` 区域展示：

- 最值得警惕的漏洞。
- 产品边界风险。
- 品牌风险。
- 执行风险。
- 自我欺骗风险。
- 下一轮必须验证的问题。

## 9.4 搜索轨迹

搜索轨迹是这个网站区别于普通多智能体聊天页的关键。

界面要展示：

- 哪些 `StrategyFamily` 已被试过。
- 哪些 family 还处于 cold start。
- 当前轮次更像 exploration 还是 refinement。
- family 随轮次积累了哪些 qualitative memory。
- 哪一轮之后某个 family 被继续探索。

当前版本允许显示内部搜索 reward、mean reward 和 UCB score，但必须明确标成：

> 沙盘内部搜索信号，不等于真实购买率、复购率或市场预测。

第一版已经在右侧 Family 轨迹面板显示这些内部指标。展示时要同时保留：

- family 当前状态词，例如已探索、值得继续、证据混合、风险偏高、尚未充分验证。
- UCB selection 的 `ucbScore` 和选择原因。
- search update 的 `reward`、`positiveUtility`、`riskPenalty`。
- family arm 更新前后的 `pullCount`、`rewardSum`、`meanReward` 和 `lastSelectedRound`。
- reward 的正向拆解、风险拆解、cap 和 mapping note。
- 明确边界文案：这些数字只解释搜索过程，不代表真实消费者购买或真实市场表现。

## 10. 当前类在网站里的视觉映射

| 当前类或对象 | 网站里的表现 | 用户能看到什么 |
|---|---|---|
| `MarketingSandbox` | 整场运行的导演 | 轮次状态、开始暂停、历史 |
| `DecisionAgent` | 中央策略家 | 当前策略和动作变化 |
| `ConsumerAgent[]` | 围在周围的人群代理 | 第一印象、购买态度、复购和竞品反应 |
| `FeedbackSynthesizer` | 汇总反馈的角色 | 整体策略感觉和下一轮方向 |
| `CriticAgent` | 专门挑刺的角色 | 风险、漏洞、待验证问题 |
| `ActionSpace` | 中央策略桌上的动作边界 | 能动哪些营销按钮 |
| `OutputContractGuard` | 运行契约保护 | 输出越界时的错误和暂停 |
| `UCBSearchController` | family 搜索轨迹 | 探索哪些策略方向、如何继续 |
| `RewardMapper` | 内部搜索信号映射 | 默认不拟人化，必要时显示“搜索信号已更新” |

## 11. 实时事件设计

要让搜索过程真的“可视化”，前端不能只等最终 `SandboxResult`。

后端后面最好能把运行过程拆成事件流，前端按事件驱动动画和面板更新。

## 11.1 建议事件

| 事件 | 前端表现 |
|---|---|
| Run started | 场景从待机进入搜索 |
| Round started | 中央策略台切换轮次 |
| Strategy family selected | family 光路点亮 |
| Strategy proposed | `DecisionAgent` 出招 |
| Consumer feedback ready | 对应消费者角色发言 |
| Feedback summary ready | 反馈光线汇总到总结者 |
| Critique ready | 批评者亮起风险提示 |
| Search updated | 搜索轨迹向下一轮更新 |
| Round completed | 时间线落下一个轮次节点 |
| Stop requested | 页面进入“正在停止”，本地 runner 不再派发下一步搜索 |
| Run stopped | 标记停止位置，并弹出是否存档 |
| Archive saved | 时间线与本地存档列表出现该档案 |
| Archive imported | 配置、轮次历史和可回放事件恢复到页面 |
| Run completed | 进入结果比较与回看状态 |
| Contract or runtime error | 场景暂停并展示错误说明 |

## 11.2 实时交互节奏

演示节奏不能快到看不清。

建议支持三种播放模式：

| 模式 | 用途 |
|---|---|
| 自动运行 | 自己看完整搜索 |
| 演示慢速 | 给老师同学讲解 |
| 单步推进 | 逐个看策略、消费者、总结者、批评者 |

这里要区分两个动作：

- `暂停` 只暂停前端播放或演示节奏，方便用户看当前事件。
- `停止` 是正式终止本地搜索运行，要把 stop 信号发给后端 runner，不再继续派发后续策略生成、消费者反馈和 UCB 更新。

真实模型调用可能已经在路上。第一版停止时可以采用协作式取消：

- 后端先停止调度新的搜索步骤。
- 如果当前模型调用支持中断，则尽量中断。
- 如果当前调用不能立刻中断，页面显示“正在停止”，等本地 runner 收到安全停止点后进入 stopped 状态。

## 12. 用户操作流程

## 12.1 一次完整使用

1. 打开网站，进入沙盘工作台。
2. 配置 API 与模型路由，或者选择示例回放模式；如果已有存档，也可以先导入存档。
3. 填写或载入产品事实、品牌事实和营销目标。
4. 选择消费者人群卡和场景卡。
5. 检查动作集与搜索设置。
6. 点击开始。
7. 看 `DecisionAgent` 选择策略并出招。
8. 看不同消费者代理依次反馈。
9. 看反馈总结者和批评者收束本轮证据。
10. 看搜索轨迹进入下一轮。
11. 暂停、回看、比较不同轮次，或者点击停止结束本地搜索。
12. 如果停止，选择是否把当前安全进度存成存档。
13. 最后导出策略讨论结果、关键风险和搜索过程摘要。

## 12.2 演示时的一条推荐路径

课堂演示建议：

1. 先展示已配置好的沙盘场景。
2. 点开 API 与模型映射，让观众知道不是所有角色都必须是同一个模型。
3. 展示人群卡、场景卡和动作集边界。
4. 运行一轮慢速搜索。
5. 指出消费者反馈和批评者反馈的区别。
6. 再运行到后续轮次，展示 family 搜索轨迹发生变化。
7. 回看一轮“看起来吸引首购但竞争韧性弱”的策略。
8. 最后展示当前值得继续验证的策略方向。

## 12.3 停止、存档和导入流程

真实搜索运行时，顶部状态栏要有一个明确的 `停止搜索` 按钮。  
它不是把画面暂停在当前帧，而是要求本地后端停止继续搜索。

建议流程：

1. 用户点击 `停止搜索`。
2. 页面提示当前 run 正在停止，并锁住会继续派发搜索的按钮。
3. 后端 runner 停止后返回 stopped 状态、停止位置和可存档快照。
4. 页面弹出选择：
   - `存档当前进度`
   - `不存档`
5. 用户点击存档后，本地生成一个可选择的档案。
6. 下一次进入网站，用户点击 `导入存档`，从本地档案列表选择一个。
7. 导入后可以先回看历史；如果存档包含可恢复搜索状态，再从安全边界继续搜索。

第一版不应假装任意中断点都能无损续跑。  
如果用户在一轮中途停止：

- 存档要保留已经完成的轮次。
- 当前轮已经通过守门和保存的事件可以回看。
- 未形成完整 observation 的部分要标成 `partial` 或 `stopped`。
- UCB 状态只能从已完成且合法的 search update 恢复，不能把半轮反馈偷偷算进搜索状态。

如果真实运行时 API 断联：

- 不要在一次短暂抖动后立刻把 run 判死。
- runner 应先进入断联等待状态，并保留当前安全进度。
- 断联超过可配置容忍时间后，自动生成一份安全存档。
- 自动存档要标清 `auto_archived` 状态和 `api_disconnect_timeout` 来源，方便导入时解释为什么中断。
- 自动存档同样只能恢复到最近安全边界，不能把未完成调用或半轮搜索结果写进 UCB 状态。

## 14. 回看与比较设计

## 14.1 回放

用户点击某一轮后：

- 三维场景恢复到该轮的主要事件序列。
- 当前策略桌面恢复该轮动作。
- 右侧恢复该轮反馈、总结、批评。
- 时间线标出该轮之前和之后搜索状态的变化。

## 14.2 比较

建议能比较：

- 两轮具体策略。
- 两个 `StrategyFamily`。
- 同一策略在不同 scenario 下的反应。
- 同一 persona 对不同策略的反应。

## 14.3 导出

第一版导出不必过度复杂，优先支持：

- 搜索过程摘要。
- 当前推荐策略方向。
- 被淘汰或暂缓策略方向。
- 消费者反馈摘要。
- 风险与待验证问题。

## 14.4 本地存档设计

存档的目的不是只保存一张截图，而是让用户以后能重新打开这次沙盘运行。

建议第一版把档案存在本地后端管理的 archive store 中，页面只负责：

- 请求保存。
- 显示存档列表。
- 导入选中的存档。
- 提示能否继续搜索。

每份存档至少保存：

| 内容 | 用途 |
|---|---|
| 档案元数据 | 名称、创建时间、停止位置、产品/项目摘要 |
| 非敏感运行配置 | 轮次设置、模型路由名、动作集、family 配置 |
| 产品、人群和场景快照 | 保证回看时背景事实一致 |
| 事件日志 | 恢复三维回放、说话泡泡和时间线 |
| 已完成轮次结果 | 恢复策略、反馈、总结、批评和搜索轨迹 |
| 搜索状态快照 | 允许从安全边界继续 UCB 搜索 |
| 存档状态 | `completed`、`stopped`、`partial` 或 `error` |

存档默认不保存：

- API Key 明文。
- 其他 provider secret。
- 不该暴露给导出材料的私密日志。

导入时应有两种入口：

- `导入回看`：恢复画面、配置快照和历史事件。
- `导入并继续搜索`：在用户重新提供可用 API 配置后，从最近安全边界继续。

真实运行还要支持断联保护：

- API 断联先给一个短暂容忍窗口，避免瞬时网络波动制造大量档案。
- 超过容忍窗口后，由本地后端自动存档已确认安全的运行进度。
- 页面要说明这是“断联自动存档”，并提示重新提供可用 API 配置后再继续搜索。

## 15. 关键视觉交互细节

## 15.1 摄像机

默认摄像机：

- 轻微俯视中央策略台。
- 能同时看到中央决策者和外围消费者。
- 不要频繁强行转镜头，避免观众迷失。

可提供：

- 总览。
- 聚焦当前发言者。
- 聚焦策略桌。
- 聚焦搜索轨迹。

## 15.2 对话泡泡

泡泡要求：

- 固定跟随 agent，不遮挡脸。
- 长度限制。
- 当前发言泡泡更亮，历史泡泡很快淡出。
- 点击泡泡能在右侧打开完整反馈。

## 15.3 数据与 3D 的关系

原则是：

> 3D 告诉用户“现在发生了什么”，HUD 告诉用户“这件事具体意味着什么”。

不要把大量长文直接铺在场景中。

## 17. 实现方式与小任务拆解

## 17.1 推荐实现方式

这个网站建议按三层实现：

| 层 | 主要职责 | 推荐形态 |
|---|---|---|
| 可视化前端层 | 三维场景、配置表单、事件播放、时间线和比较视图 | Web 前端 + Three.js 场景 |
| Web 运行桥接层 | 接收配置、保护 API 密钥、启动沙盘、把过程拆成事件流 | Python Web 后端包住现有沙盘 |
| 沙盘核心层 | `MarketingSandbox`、各 Agent、`ActionSpace`、UCB 搜索和结果对象 | 保持当前 Python 代码职责 |

第一版的关键不是立刻把所有真实模型调用都接进 3D，而是先把两条路径打通：

1. 用示例事件流驱动前端，确保三维沙盘、面板和时间线都能演示。
2. 用后端事件桥接真实 `MarketingSandbox`，把真实策略、反馈、批评和搜索更新逐步替换示例事件。

## 17.2 推荐开发顺序

建议按下面三个里程碑推进：

| 里程碑 | 目标 | 完成标志 |
|---|---|---|
| M1 离线可看 | 不接真实模型也能看见沙盘演示 | 示例回放能驱动 3D agent、泡泡、右侧面板和时间线 |
| M2 真实可跑 | 网站能启动、停止并保存真实沙盘搜索 | 配置 API 后能收到真实轮次事件，停止本地 run，并保存可导入档案 |
| M3 课堂可讲 | 运行稳定、节奏可控、可回看和可恢复 | 慢速演示、单步推进、存档导入、错误提示和关键摘要都可用 |

## 17.3 小任务清单

下面按推荐执行顺序拆任务。每个任务都尽量小到能单独完成、单独验收。

## 19. 设计边界

## 19.1 数字边界

网站可展示：

- 策略动作中的价格、折扣、券额和预算配置。
- 内部搜索使用的 family trace、reward breakdown、mean reward 和 UCB score。
- 运行轮次和搜索状态。

网站不能误导性展示：

- 消费者代理生成的购买概率。
- 复购概率。
- 市场占有率预测。
- 把搜索 reward、mean reward 或 UCB score 当真实市场表现。

## 19.2 角色边界

- `DecisionAgent` 是出策略的人。
- `ConsumerAgent` 是人群反应，不是市场预测器。
- `FeedbackSynthesizer` 是总结者，不是打分器。
- `CriticAgent` 是风险和漏洞发现者，不是随口报损益的人。

## 19.3 演示边界

课堂演示时应明确：

> 这个系统用于比较营销策略、暴露人群反应差异和风险假设，不替代真实调研和真实市场验证。

## 21. 当前推荐结论

这个网站最有辨识度的形态不是“营销仪表盘加一点 3D 装饰”，而是：

> 一个可观看的实时营销策略议事场。

中央小人持续出招，周围人群持续回应；每一次反馈、总结和批评都在推动搜索轨迹变化。用户既能沉浸地看到 agent 在沙盘里互动，也能清楚地读到这轮营销策略到底动了什么、为什么被继续探索、为什么仍然需要真实市场验证。
